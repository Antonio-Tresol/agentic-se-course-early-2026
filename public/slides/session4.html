Towards Agent-First SE

When teams of agents become the primary code producers, the engineering work shifts from writing code to building the system around the code. The industry calls this harness engineering. This session is the shape of that work, drawn from the seven readings.

Part 1 · The Shift

Reading the Frontier Labs Honestly

Anthropic, OpenAI, and Google are publishing a lot here. Hold two things at once.

Discount the headline

Frontier labs have strong commercial incentives to say AI can automate coding right now. The framing is calibrated to investors, customers, and the model narrative. Take any blanket claim with a grain of salt.

Take the practical seriously

The arms race between labs makes it expensive to lie to themselves about their own codebases. What they describe operating internally is probably real, even if its public presentation is glossier than the lived experience.

Crucially, the OpenAI piece is honest about the conditions: "There was no pre-existing human-written code to anchor the system. From the beginning, the repository was shaped by the agent." They explicitly disclaim generalisation. What they actually argue for, once you strip the framing, is mundane: tests, types, lints, reviews, clean documentation, sound architecture, security. Reasonable. Actionable. Not magic.

Part 2

Six Pillars

Things multiple sources converge on, with the actual passages attached. None is novel as an engineering practice. What is novel is treating them as the centre of the work, not its periphery.

Pillar 01

The Harness IS the Engineering Work

The model is roughly constant in any given week. The codebase, the tools, the docs, the test infrastructure, the way work is described and decomposed: that is what you control. That is where leverage lives.

"Building software still demands discipline, but the discipline shows up more in the scaffolding rather than the code. The tooling, abstractions, and feedback loops that keep the codebase coherent are increasingly important."

OpenAI · Harness Engineering · Feb 2026

Lopopolo pushes this further in the talk: "Loadbearing infrastructure for our AI future: you can just prompt things." AGENTS.md is a prompt. Lint error messages with remediation are prompts. Review agents that comment on PRs are prompts. The repo is a giant standing prompt the agent re-reads every run.

Pillar 02

Persistent State, Not Chat

A long-running agent treated as a chat session forgets, drifts, and contradicts itself. The same agent treated as a worker that reads and writes durable artefacts is a different animal.

"Each new session begins with no memory of what came before."

Anthropic · Effective Harnesses for Long-Running Agents · Nov 2025
Anthropic harnesses

claude-progress.txt, JSON feature lists, init.sh as the standard startup. Each session re-reads the artefacts.

OpenAI harness

A structured docs/ tree as the system of record. Plans treated as first-class artefacts, versioned alongside the code they describe.

Pillar 03

Legibility for Agents

What is invisible to an agent is invisible to the system. Make the implicit explicit.

"From the agent's point of view, anything it can't access in-context while running effectively doesn't exist. That Slack discussion that aligned the team on an architectural pattern? If it isn't discoverable to the agent, it's illegible in the same way it would be unknown to a new hire joining three months later."

OpenAI · Harness Engineering · Feb 2026

Lopopolo in the talk: "Code in the file system is also text which means it's effectively prompts you're giving to your coding agent." Practical consequences: AGENTS.md as a roughly hundred-line table of contents, structured docs/ folders, one canonical way to write async helpers, one programming language, one way of writing CI scripts. Same shapes everywhere reduce the attention the model has to spend understanding any one place.

Pillar 04

Mechanical Signal Everywhere

Agents produce output faster than humans can carefully read it. The harness has to provide signal a machine can check.

"By enforcing invariants, not micromanaging implementations, we let agents ship fast without undermining the foundation. In a human-first workflow these rules might feel pedantic. With agents, they become multipliers: once encoded, they apply everywhere at once."

OpenAI · Harness Engineering · Feb 2026

Custom lints with remediation

Error messages written to inject remediation instructions back into the agent's context. Lints become prompts.

Tests about the source

Not just behavioural tests. Lopopolo: a test that asserts files are no longer than 350 lines, structural tests on dependency edges between layers.

Pillar 05

Sandboxes and Observable Execution

Isolated execution per unit of work, with the agent able to inspect logs, metrics, and traces directly.

Per-worktree app

OpenAI: every change runs against a fully isolated app instance, with an ephemeral observability stack (logs, metrics, traces) torn down with the worktree. Codex queries logs with LogQL and metrics with PromQL.

Per-issue workspace

Symphony's three safety invariants are explicit: the coding agent runs only inside its workspace, the workspace is contained inside a sanitised root, and the workspace key uses only safe characters.

Same shape across the sources: an isolated, replayable, observable place for the agent to act and a stable boundary against everything else.

Pillar 06

Multi-Agent Coordination Needs Structure

A single agent on a task does not need orchestration. A team of agents on a shared codebase does. The sources agree on this and disagree on the shape, which is the next section.

A Anthropic harnesses. Two agents, an initialiser and a coding agent.
B OpenAI harness. A "Ralph Wiggum Loop": the agent reviews its own changes, then requests other agents to review, locally and in the cloud, until satisfied.
C Symphony. Many parallel agents pulling work from a Linear board.
D Anthropic Agent Teams. Three to five peer teammates with a shared task list and direct messaging.
Part 3

Approaches

Six instantiations of the pillars. Same problem, different pressure points. Read them as a sample of what the field has tried, not as competing methodologies you have to pick from.

Approach 01 · Anthropic · Nov 2025

Effective Harnesses for Long-Running Agents

A two-agent pattern. An initialiser establishes a structured environment. A coding agent makes incremental progress and leaves clean artefacts.

A JSON feature list with pass and fail status, so the system always knows what is done.
claude-progress.txt as a session-by-session log, read at the start of every new session.
init.sh standardising dev-server startup. Browser automation for end-to-end verification.
"Clean state" defined as "the kind of code that would be appropriate for merging to a main branch."

Tradeoffs they call out: single vs multi-agent unclear, browser automation cannot catch every bug, generalisation beyond web dev unknown.

Approach 02 · OpenAI · Feb 2026

Harness Engineering: Codex in an Agent-First World

Five months. One internal product. Roughly one million lines, fifteen hundred PRs, three to seven engineers. Zero manually-written code. Started from an empty git repository.

The codebase is the harness

AGENTS.md as a hundred-line table of contents. A structured docs/ tree as the system of record. Plans as first-class artefacts. A "doc-gardening" agent opens fix-up PRs against stale documentation.

Application legibility

App bootable per git worktree, so Codex drives one isolated instance per change. Chrome DevTools Protocol wired into the agent runtime. Local observability stack (Vector to Victoria Logs / Metrics / Traces) ephemeral per worktree.

Layered domain architecture

Each business domain split into Types to Config to Repo to Service to Runtime to UI. Cross-cutting via a single Providers interface. Enforced mechanically with custom Codex-generated lints and structural tests.

Throughput changes the merge philosophy

Minimal blocking gates. Short-lived PRs. Test flakes addressed with follow-up runs rather than blocking. Justified because corrections are cheap and waiting is expensive.

Approach 02 (continued) · The Talk

The Operational Gloss

Lopopolo's AI Engineer London talk presents the same nine months of work with sharper practical detail.

A Friday garbage-collection day. One day a week, every engineer takes the slop they observed in the last week and figures out how to categorically eliminate it. Comments get bucketed by persona (front-end architect, reliability, scalability). Each persona becomes a review agent that runs on every push.
B Tests about the source code. A test that asserts no file is longer than three hundred and fifty lines. Structural tests on dependency edges between layers. Code shape becomes a load-bearing concern, not an aesthetic one.
C Plan mode opinion. "I should be able to drop a ticket in and have it do the job anyway." If you do use plans, push them as separate PRs and human-review every line; an unread approved plan is a stack of bad instructions.
D The closing principle. "Every time I have to type continue to the agent is a failure of the harness to provide enough context around what it means to continue to completion."
Approach 03 · OpenAI · Apr 2026

Symphony: Ticket Boards as the Control Plane

Once the codebase was harness-ready, the next bottleneck appeared: humans context-switching across multiple Codex sessions.

"We had effectively built a team of extremely capable junior engineers, then assigned our human engineers to micromanaging them. That wasn't going to scale."

OpenAI · Symphony · Apr 2026
A Linear as a state machine. Every open issue maps to a dedicated agent workspace. Symphony continuously watches the board and ensures every active task has an agent in the loop.
B Workflow lives in WORKFLOW.md. The team's workflow is repo-versioned policy. Symphony itself is a SPEC.md, language-agnostic. The reference implementation is in Elixir.
C Objectives, not transitions. They explicitly moved away from rigid state machines. "Models can solve bigger problems than the box we try to fit them in."
Approaches 04 / 05 / 06

Three More Shapes

04 SASE · Hassan et al., Sep 2025. An academic decomposition. Four redefined SE pillars (Actors, Processes, Artefacts, Tools) plus a structured duality of SE for Humans and SE for Agents. Six engineering activities (BriefingEng, ALE, ATME, AGE, ATLE, ATIE). Useful for naming the parts when the practitioner literature gets confusing.
05 Verified Spec-Driven Development · Doll, Mar 2026. Spec Supremacy: the spec is the highest authority below the human. Four-Dimensional Convergence: spec, tests, implementation, and formal proofs all survive an Adversary running with fresh context and zero tolerance.
06 Agent Teams · Anthropic, experimental in Claude Code 2.1.32+. A team lead spawns three to five peer teammates with their own context windows. They share a task list and message each other directly via a mailbox. Hooks (TaskCreated, TaskCompleted, TeammateIdle) gate quality.

Part 4

Open Debates

Four places the sources actually disagree, with the passages that put each side on the record.

Greenfield versus brownfield

OpenAI is explicit: the work "should not be assumed to generalize without similar investment, at least not yet." Anthropic and SASE are aimed at hybrid teams in existing organisations. The asymmetry is real and acknowledged.

State machines versus objectives

Symphony explicitly moved away from rigid transitions because "models can solve bigger problems than the box we try to fit them in." Verified Spec-Driven Development relies on a rigid pipeline of phases. Reasonable people pick differently.

Specs versus exploration

VSDD's Spec Supremacy says nothing exists without spec traceability. The OpenAI harness says: keep the codebase navigable so agents can explore. gitcnd, replying to the VSDD gist: "The spec becomes airtight because you built it, not before you build it."

Merge gates and human review

OpenAI removed most blocking gates and ships agent-to-agent reviews. SASE proposes structured Merge-Readiness Packs. VSDD adds Four-Dimensional Convergence including formal proofs. Same goal, very different cost structures.

Closing

The Bottleneck Is the System

Coding agents are not magic. The argument the seven readings make, with different vocabulary, is that the engineering work moves up the stack: you stop optimising the writing of code and you start optimising the system that makes correct code easy to produce. That system is your harness. The capstone is a chance to build one.
