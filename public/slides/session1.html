Part 1

How Claude Code Works

Three phases that blend together, powered by a model and a set of tools.

code.claude.com/docs/en/how-claude-code-works

The Agentic Loop

The phases blend. Each tool call informs the next. You can interrupt at any point.

Claude Code is an Agentic Harness

The loop is powered by two components.

Models

Claude reasons about your code, plans steps, and picks the next action. Sonnet handles most tasks; Opus for complex reasoning. Switch with /model.

Tools

Without tools, Claude can only respond with text. With tools, it can read files, edit code, run commands, search the web, call external services.

Claude Code provides the harness around Claude: the tools, the context management, and the execution environment that turn a language model into a capable coding agent.

What Claude Has & What Claude Sees

Built-in tool categories
File operations. Read, edit, create, reorganize.
Search. Glob patterns, regex, codebase exploration.
Execution. Shell commands, servers, tests, git.
Web. Search, fetch docs, look up errors.
Code intelligence. Type errors, definitions (via plugin).
In your working directory, Claude accesses
Your project files and subdirectories
Your terminal and any command you can run
Your git branch, diff, and history
CLAUDE.md and auto memory
Any MCP servers and skills you configured

Part 2

Extending Claude Code

The built-in tools cover most coding tasks. The extension layer is how you customize what Claude knows, what it can reach, and what runs automatically.

code.claude.com/docs/en/features-overview

Seven Ways to Extend

CLAUDE.md

Persistent context loaded every session. Always-on rules and conventions.

Skills

On-demand knowledge and invocable workflows. Triggered by Claude or by /name.

MCP

A protocol for connecting Claude to external services: databases, Slack, Figma, your tracker.

Subagents

Isolated workers with their own context window. Return a summary to the main session.

Hooks

Deterministic scripts that run on lifecycle events. No LLM involved.

Plugins

Bundle skills, hooks, subagents, and MCP servers into one installable unit.

When to Reach for Each

Add features when a trigger shows up. You do not need to configure everything up front.

TriggerAdd
Claude gets a convention wrong twiceCLAUDE.md rule
You keep typing the same prompt to start a taskUser-invocable skill
You paste the same playbook for the third timeSkill for the workflow
You keep copying data from a system Claude cannot seeMCP server
A side task floods your conversation with outputSubagent
Something must happen every time, no exceptionsHook
A second repository needs the same setupPlugin

What Each Feature Costs

Every extension consumes some context. Understanding the load pattern matters more than the raw cost.

FeatureWhen it loadsWhat loads
CLAUDE.mdSession startFull content, every request
SkillsStart + on useDescriptions at start, full body on invocation
MCPSession startTool names, schemas deferred until used
SubagentsWhen spawnedFresh isolated context, returns a summary
HooksOn triggerNothing, they run externally

Keep CLAUDE.md short. Move reference material to skills. Use subagents when a task would otherwise flood your main context.

Part 3

Best Practices

Most of the patterns in this guide are downstream of one constraint.

code.claude.com/docs/en/best-practices

Give Claude a Way to Verify

The single highest-leverage thing you can do. Tests, screenshots, linters, a Bash check.

BeforeAfter
"implement a function that validates email addresses" "write validateEmail. cases: user@example.com true, invalid false, user@.com false. run the tests after."
"make the dashboard look better" "[paste screenshot] implement this. take a screenshot of the result, compare, fix differences."
"the build is failing" "the build fails with: [error]. fix it, verify the build passes, address the root cause."

Explore, Plan, Code, Commit

Four phases. Separate research from implementation to avoid solving the wrong problem.

Skip planning for small scoped changes. Use it when multiple files are involved.

Write an Effective CLAUDE.md

Loaded every session. Treat it like code: review it, prune it, check it in.

Include
Bash commands Claude cannot guess
Code style rules that differ from defaults
Test instructions and runners
Repo etiquette (branches, PRs)
Architectural decisions
Environment quirks and gotchas
Exclude
Anything Claude can read from the code
Standard language conventions
Detailed API docs (link instead)
Info that changes frequently
File-by-file descriptions
"Write clean code" style platitudes

Bloated CLAUDE.md files cause Claude to ignore your actual instructions. Run /init to bootstrap.

Common Failure Patterns

The kitchen sink session. One task, then another, then the first again. Context full of noise. Fix: /clear between unrelated tasks.
Correcting over and over. Context pollutes with failed approaches. Fix: after two failed corrections, /clear and rewrite the prompt.
The over-specified CLAUDE.md. Important rules drown in noise. Fix: ruthlessly prune.
The trust-then-verify gap. Plausible code that misses edge cases. Fix: always provide something to verify against.
The infinite exploration. Claude reads hundreds of files filling the context. Fix: scope, or use a subagent.
Part 4

Permission Modes

Default mode has you review each action. Looser modes let Claude work in longer stretches. Pick the one that matches the work.

code.claude.com/docs/en/permission-modes

Six Modes

ModeWhat runs without askingBest for
defaultReads onlyGetting started, sensitive work
acceptEditsReads, edits, common fs commandsIterating on code you review after
planReads only, writes a planExploring a codebase before changing it
autoEverything, with classifier checksLong tasks, reducing prompt fatigue
dontAskOnly pre-approved toolsLocked-down CI and scripts
bypassPermissionsEverything except protected pathsIsolated containers and VMs only

Writes to protected paths (.git, .claude, .bashrc, .mcp.json, etc.) never auto-approve in any mode.

Plan Mode & Auto Mode

Plan mode

Claude researches and proposes changes without making them. Reads files, runs shell commands to explore, writes a plan.

Enter: Shift+Tab or /plan. When ready, choose: approve and auto, approve and review, keep planning, or refine with Ultraplan.

Auto mode RESEARCH PREVIEW

A classifier model reviews each action. Blocks scope escalation, unknown infrastructure, hostile-content-driven actions.

Requires Team/Enterprise/API plan, Sonnet 4.6 or Opus 4.6, Anthropic API. Falls back after 3 consecutive or 20 total blocks.

# during a session
Shift+Tab cycle default → acceptEdits → plan
# at startup
claude --permission-mode plan
claude --enable-auto-mode
Part 5

Scheduled Tasks

Re-run a prompt on an interval. Poll a deployment, babysit a PR, check a long-running build, or set a one-shot reminder.

code.claude.com/docs/en/scheduled-tasks

/loop in Three Forms

1 Interval and prompt. /loop 5m check if the deploy finished. Fixed cron schedule.
2 Prompt only. /loop check CI and address review comments. Claude picks the interval (1 min to 1 hour) based on what it sees.
3 Nothing. /loop. Runs the built-in maintenance prompt, or your project's .claude/loop.md.

For a one-shot reminder, ask in natural language: "in 45 minutes, check whether the integration tests passed."

Under the Hood

Cron tools
CronCreate schedule a task
CronList list scheduled tasks
CronDelete cancel by 8-char ID
Limits
Up to 50 tasks per session
Recurring tasks expire after 7 days
Session-scoped, die when session exits
Fire between turns, not mid-response
# 5-field cron expressions
*/5 * * * * every 5 minutes
0 9 * * 1-5 weekdays at 9am local time
30 14 15 3 * March 15 at 2:30pm local

For durable scheduling use Cloud or Desktop scheduled tasks.

Recap

What to Remember

Claude Code is a harness. Model plus tools plus context management. The model reasons, the tools act.
Context is the constraint. Most best practices are about managing it.
Verify, then explore, plan, code, commit. Specific prompts with something to check against.
CLAUDE.md for always-on, skills for on-demand. Keep CLAUDE.md short.
Pick the mode that matches the work. Default for sensitive, plan for exploration, acceptEdits for iteration.
