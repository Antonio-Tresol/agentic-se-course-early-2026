Skills

Extend What Claude Knows

A SKILL.md file with instructions. Claude loads it when relevant, or you invoke it with /name.

Where skills live
~/.claude/skills/ personal, all projects
.claude/skills/ project, check into git
Plugin skills use plugin:name namespace
Two types of content
Reference. Knowledge Claude applies: conventions, patterns, style guides.
Task. Step-by-step instructions: deploy, review, release.

SKILL.md Anatomy

---
name: fix-issue
description: Fix a GitHub issue
disable-model-invocation: true
allowed-tools: Bash(gh *)
---

# instructions follow
Fix GitHub issue $ARGUMENTS.

1. Read the issue with gh
2. Understand requirements
3. Implement the fix
4. Write tests
5. Commit and push
Key frontmatter fields
nameBecomes the /slash-command
descriptionHow Claude decides when to load it
disable-model-invocationOnly you can trigger it
allowed-toolsPre-approve tools while active
context: forkRun in an isolated subagent
$ARGUMENTSReplaced with what follows /name

Descriptions load at start. Full body loads only when invoked.

Subagents

Isolated Workers, Own Context

A subagent does its work in a fresh context window and returns a summary. Your main conversation stays clean.

Explore

Haiku. Read-only. Fast codebase search and analysis. Three thoroughness levels.

Plan

Inherits model. Read-only. Gathers context for plan mode without nesting.

General-purpose

Inherits model. All tools. Complex multi-step tasks that need both reading and writing.

Use a subagent when a side task would flood your context with output you will not reference again.

Model Context Protocol

Connect Claude to External Tools

MCP is an open protocol for connecting AI to external services. Each server exposes tools Claude can call.

# add a server from the registry
claude mcp add slack --transport http https://mcp.anthropic.com/slack

# add a local stdio server
claude mcp add my-db -- npx -y @my/db-server --connection-string $DB_URL

# list active servers
claude mcp list

# check token costs per server
/mcp

Examples: Slack, GitHub, Jira, Figma, Notion, PostgreSQL, Sentry, browser control, custom internal APIs.

MCP Configuration

Scope hierarchy
Local. .mcp.json in project root. Gitignored. Highest priority.
Project. .claude/settings.json shared with the team.
User. ~/.claude/settings.json across all projects.
Three transports
stdio. Local process. Most common.
http. Remote, streamable HTTP.
sse. Server-sent events (legacy).
Tool search (deferred loading)

MCP servers can expose dozens of tools. Claude Code defers their full schemas and loads only tool names at session start.

When Claude needs a specific tool, it searches by keyword, loads the schema on demand, then calls it. Idle servers cost almost nothing in context.

Run /mcp to see per-server token costs and connection status.

Adapting A Coding Agent

Skills, MCP, subagents, and the deeper principles of context engineering.

Effective Harnesses for Long-Running Agents · Anthropic · Nov 2025

The Long-Running Agent Problem

Complex tasks span many context windows. Each new window starts with no memory of what came before.

One-shotting. The agent tries to build the entire app at once, runs out of context mid-feature, and leaves the next session to guess what happened.
Declaring victory early. A later agent sees some features built, assumes the job is done, and stops working.
Unmarked bugs. Features get marked complete without proper end-to-end testing. Problems compound across sessions.

Imagine a project staffed by engineers in shifts, where each new engineer arrives with no memory of what happened on the previous shift.

Two-Part Solution

A specialized first session that sets up the environment, then every subsequent session makes incremental progress.

1. Initializer agent

Runs once, on the very first session. Sets up:

Feature list. Structured JSON with all required features, each marked as failing. Gives the agent a clear definition of "done."

Progress file. claude-progress.txt for session-to-session handoff notes.

Init script. init.sh to start the dev server and run a basic smoke test.

Initial git commit. Clean starting state the agent can always revert to.

2. Coding agent

Every subsequent session follows the same routine:

Get bearings. Read progress file, git log, feature list. Run init.sh and verify basic features still work.

Work on one feature. Incremental progress, not one-shotting. Test end-to-end (browser automation, not just unit tests).

Leave it clean. Git commit with a descriptive message, update the progress file, mark the feature as passing only after real verification.

Failure Modes and Solutions

ProblemInitializer doesCoding agent does
Declares victory early Feature list with all features marked failing Read the list. Pick one feature. Work only on that.
Leaves bugs or mess Set up git repo and progress file Read progress + git log at start. Commit and update notes at end.
Marks features done without testing Structured feature list with passes: false Verify end-to-end (browser automation). Only flip to true after real testing.
Wastes time figuring out the environment Write init.sh with dev server and smoke test Run init.sh first. Fix any existing bugs before starting new work.

The key insight: let agents quickly understand the state of work by combining a progress file with git history. Inspiration came from knowing what effective human engineers do every day.
