Reframe

Closing a Gap on Purpose

A greenfield task is small enough to fit in the model. A brownfield task is bigger than any context window. The discipline we are about to unpack is how to close that gap deliberately.

Context engineering is the art and science of curating what will go into the limited context window from that constantly evolving universe of possible information.

Anthropic, Effective Context Engineering for AI Agents (May 2025)
Part 1 of 3

Invariants

Six things every source in this week's reading agrees on, even where they disagree about the concrete workflow.

Invariant 1 of 6

Context Is the Only Lever

LLMs are stateless. Every turn, the only thing shaping the next token is the tokens already in the window. Better tokens in, better tokens out.

Priority order for what goes in
1Correctness. Wrong facts are worst. The agent builds on them.
2Completeness. Missing facts cascade into wrong answers.
3Size and noise. Everything else clutters reasoning.
Trajectory matters too

If the conversation shows the agent doing wrong things and being corrected, the next most likely token is another wrong thing. The history is part of the prompt.

Output quality is roughly (correctness × completeness) divided by (noise + size).

Invariant 2 of 6

Context Is Scarce

A full context window is not a smart context window. Attention is finite, and it thins out as the window fills.

As the number of tokens in the context window increases, the model's ability to accurately recall information from that context decreases.

Anthropic, on context rot

The more you use the context window, the worse outcomes you will get.

Geoffrey Huntley, via ACE-FCA

Invariant 3 of 6

Compact on Purpose

Instead of steering one long conversation until it breaks, compress the useful tokens into a Markdown file, review it, and start again. The artefact persists; the chat does not.

Compaction is the practice of taking a conversation nearing the context window limit, summarising its contents, and reinitiating a new context window with the summary. Preserve architectural decisions and unresolved issues; discard redundant outputs.

Invariant 4 of 6

Sub-Agents Are Context Control, Not Roles

Not a product manager, a QA engineer, and a frontend specialist in a trench coat. A fresh context window that goes looking for something specific and returns a short answer.

Each sub-agent does the heavy reading, grepping, and file-opening in its own window. The parent receives one tight paragraph, typically one to two thousand tokens. The parent never paid for the search.

Invariant 5 of 6

Leverage Moves Upstream

Human attention is a scarce resource too. Put it where it bends the whole downstream curve.

This is also why code review is really about mental alignment, not bug-catching. With agents writing two to three times the volume, nobody reads every diff. The artefacts further upstream are shorter, higher signal, and arrive before any code is written.

Invariant 6 of 6

Enforce Mechanically, Not With Words

Instructions in a prompt compete with other instructions and can be silently skipped. A failing test, a linter, a CI gate, or a pre-commit hook cannot. When you can move a rule from words into code, do it.

Why words leak

Frontier models follow roughly 150 to 200 instructions with consistency. By the time you add CLAUDE.md, the system prompt, tool descriptions, MCP instructions, and a workflow prompt, you are rolling the dice on which rules get attended to.

Don't use prompts for control flow if you can use control flow for control flow. The if-statement is really, really powerful.

Dex Horthy, No More Slop
Mechanical backstops to prefer
Verifiable feedback loops. Tests that fail on regression, not vibes.
Structured artefacts. A feature list with passes: false until end-to-end tests flip it.
Environment scripts. init.sh proves the dev loop works before new work starts.
Review gates. Human must sign off the plan file before implementation runs.
Hooks and linters. Block bad commits without asking the agent to remember.

Part 2 of 3

Approaches

Four concrete instantiations of the six invariants, from different vendors and authors, drawn from this week's reading.

Approach 1 · Anthropic · May 2025

The Abstract Frame

The earliest and most general articulation. Not a workflow, a vocabulary. Everything that follows can be read as a concrete instantiation of these ideas.

Just-in-time retrieval

Agents hold lightweight identifiers (file paths, queries, URLs) and load data into context on demand, rather than pre-loading everything.

Progressive disclosure

Each interaction yields context that informs the next decision. The agent discovers what it needs through exploration, not prebriefing.

Compaction

When nearing the limit, summarise and reinitiate. Keep architectural decisions and open issues. Drop redundant tool output.

Sub-agent architectures

Specialised agents handle focused tasks in clean context windows, returning roughly one to two thousand tokens of distilled summary.

Approach 2 · HumanLayer · December 2025

Research, Plan, Implement

Frequent intentional compaction, organised as three phases. Each phase lives in its own context window and writes its output to a Markdown file.

Research

Compress truth. Objective facts about how the codebase works today. No implementation opinions. Ticket hidden from the researcher.

Plan

Compress intent. Exact files, exact changes, code snippets, test checkpoints. Human reviews before any code is written.

Implement

Execute. Read the plan, do the work, compact progress back after each phase. The file is the source of truth, not the chat.

A bad line of code is a bad line of code. A bad line in a plan is a hundred bad lines of code. A bad line in research is thousands.

ACE-FCA
Approach 3 · HumanLayer · February 2026

CRISPY QRSPI

Eight stages instead of three. Same principles; the planning monolith is split so fewer instructions live in each prompt and human review happens earlier.

Why more stages? Instruction budget. The December RPI prompt had 85 directives in one prompt. Splitting them across stages keeps each prompt under the model's reliable instruction-following limit, and gives the human a shorter artefact to redirect at each step.

Approach 4 · Google · December 2025

Conductor

A Gemini CLI extension that ships the same shape under different vocabulary. Three slash commands. The specs and plans are Markdown files in the repository, not chat history.

/conductor:setup captures product goals, technical stack, and team workflows into repository artefacts.
/conductor:newTrack generates specs and phased plans (phases, tasks, subtasks). Reviewed before implementation.
/conductor:implement executes the plan with progress saved to files. Pause and resume from the artefacts.

Control your code. Shift project context from chat windows into the codebase itself.

Google Developers Blog, on Conductor

Part 3 of 3

Open Debates

Four places the sources disagree, including places where the same author disagrees with himself two months later. None of this is settled.

Debate 1 · where does human review pay?

Read the Plan, or Read the Code?

ACE-FCA, December 2025

"A well-written 200-line plan is more reviewable than 2,000 lines of generated code. Reviewing plans maintains team alignment better than post-implementation code review."

No More Slop, February 2026

"Don't read the plans. Please read the code. Plans hide surprises and you end up reviewing the same feature twice."

Same author, two months apart. What changed? Possibly: model capability, workflow maturity, or the realisation that plans create a false sense of safety when humans skim them. The honest answer is that we do not know yet.

Debate 2 · how much ceremony

When Is the Full Pipeline Overkill?

The ceiling of problems you can solve goes up with how much context engineering you are willing to do. But running the full pipeline on a one-line change is its own kind of waste.

TaskWorkflow
Tiny change. Rename, button colour, typo.Just talk to the agent.
Small feature in one file.Short plan, no dedicated research.
Medium, across multiple files.Research, plan, implement.
Hard, cross-cutting, unfamiliar.Full pipeline. Sub-agents. Human review at every compaction.

It takes reps. You will over-engineer simple tasks and under-engineer hard ones. The calibration does not come from a slide.

Debate 3 · how to shape work

Vertical Slices or Horizontal Layers?

Left untreated, models write horizontal plans: all the database, then all the services, then all the API. Two thousand lines later, nothing runs end-to-end.

Horizontal plan

Phase 1: all DB migrations. Phase 2: all services. Phase 3: all endpoints. Phase 4: UI. Test at the end.

Vertical plan

Phase 1: one slice end-to-end with mock data. Phase 2: wire the service layer. Phase 3: swap mocks for real DB. Test at every seam.

Vertical slices produce natural checkpoints. But not every problem factors this way: some architectural refactors are genuinely horizontal, and forcing verticality produces awkward scaffolding. Judgement call.

Debate 4 · trust and verification

How Far Can the Agent Verify Itself?

Mechanical enforcement catches a lot. It does not catch everything. Where is the boundary?

Things machines verify well
Unit and integration test passes
Linter, type-check, build green
End-to-end browser automation on known flows
Structural invariants enforced by the codebase
Things that still need humans
Is this the right abstraction for the team?
Does the architecture still hang together?
What fails silently in production next quarter?
Are we solving the problem the user asked for?

Next week's readings push on this boundary further: long-running harnesses, spec-driven verification, and agent teams all try to move the line.

Closing

Nothing Here Is Settled

The invariants feel stable. The approaches are iterating in public. The debates are real. Your job as a practitioner is not to pick one workflow and defend it, but to run experiments and recognise which invariant a new practice is serving.

1 Adopt the invariants. They survive the turnover in workflows.
2 Pick one approach and get reps. Do not min-max across tools.
3 Move rules from words to code whenever you can. Hooks, linters, tests, structural gates.
4 Share what breaks. The field is learning in public.
